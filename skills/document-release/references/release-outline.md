# Release outline

Use this when the release needs a more structured pass.

## Core facts

- What version, feature, or change set is this release about?
- Which PRs, issues, or commits define the shipped scope?
- Is any part of the work gated, staged, or only partially rolled out?

## Audience questions

- What should users notice immediately?
- What does support or ops need to watch for?
- Is there any action required after rollout?
- What misconception is most likely if the note is written too vaguely?

## Safety checks

- Are limitations and exclusions clearly stated?
- Does the note accidentally promise future work as present functionality?
- Does the note mention migration or rollback implications when relevant?
- Can a reader understand the change without reading the code diff?
