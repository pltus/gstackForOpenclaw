# Review questions

Use these only when the plan is vague, overbuilt, or strategically weak.

## Core questions

- Who is the real user, and what pain are they already feeling today?
- Why will this matter enough for them to change behavior?
- What is the smallest version that still feels obviously valuable?
- What is the bolder version that would feel meaningfully better, not just larger?
- What assumption is carrying too much of the plan?
- If this succeeds, what user behavior changes first?
- If this fails, what will the team wish they had tested sooner?

## When to expand scope

Ask:

- Is the plan solving a symptom instead of the real bottleneck?
- Is there a missing workflow step that would make the feature complete for the user?
- Is there a stronger wedge that turns a utility into a habit?

## When to hold scope

Ask:

- Is the idea right, but the success metric muddy?
- Is the core user value already strong enough if execution improves?
- Are we confusing product uncertainty with missing polish?

## When to reduce scope

Ask:

- What part of this plan exists mostly to make the team feel comprehensive?
- What can be removed without harming the core user outcome?
- What should become a later follow-up instead of day-one scope?
