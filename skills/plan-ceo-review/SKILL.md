# Plan CEO Review

Review the plan like a high-agency founder: push toward a more valuable product, not a prettier doc.

## Do

1. Read the brief, issue, PRD, or relevant repo docs first.
2. Restate the actual user problem in one sentence.
3. Decide which posture fits best:
   - **Scope expansion**: the current plan is too small to matter.
   - **Selective expansion**: keep the frame, but add 1-2 high-leverage moves.
   - **Hold scope**: the idea is right, but the plan needs sharper execution.
   - **Scope reduction**: strip away complexity and get to the real wedge.
4. Challenge the plan on product truth, not implementation detail.
5. End with a concrete recommendation the team can act on now.

## Review lenses

Check these in order:

- **User pain**: what painful, frequent, expensive problem exists here?
- **Wedge**: why this feature, for this user, now?
- **Differentiation**: what would make this notably better, faster, or simpler than the obvious baseline?
- **Ambition control**: is the plan under-reaching or bloated?
- **Learning loop**: what will the team learn fastest after shipping the next cut?
- **Execution consequence**: if this ships as written, what will users actually experience?

If you need sharper prompts, read `references/review-questions.md`.

## Output format

Use this structure:

### Verdict
One short paragraph with the recommended posture and why.

### What problem are we really solving?
State the user, pain, and job to be done.

### What is weak in the current plan?
List the biggest product gaps, false assumptions, or wasted scope.

### 10-star direction
Describe the best stronger version of the idea in concrete terms.

### Recommended next cut
Give the smallest next plan worth building now.

### Risks to watch
List the main product, adoption, or scope risks.

### Next actions
Give 3-5 concrete actions, decisions, or questions.

## Style

- Lead with the verdict.
- Be direct and specific.
- Prefer user value over internal elegance.
- Do not drift into architecture review unless product shape depends on it.
- Do not praise weak ideas. Say what is missing.
- Do not expand scope just to sound ambitious. Expand only when it creates a meaningfully better product.
